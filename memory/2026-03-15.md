## OOM Crash & Hard Reboot (~8:29 AM MST)

### What Happened
1. 08:29:33 — OOM kill: python3 (pid 3374158) from claw-kanban.service — 13.3 GB RSS
2. 08:33:30 — OOM kill: python3 (pid 3373705) from claw-kanban.service — 10.3 GB RSS
3. 24 GB combined from kanban builder processes on 32 GB machine → cascading OOM
4. systemd-journald and systemd-resolved crashed → unresponsive → hard reboot at ~08:35
5. Also: nq-tournament.service crash-looping (101,176 restarts!) with "Failed to determine supplementary groups" — `User=rob/Group=rob` in a user-level unit (not allowed)

### Fixes Applied
1. **claw-kanban.service** — added `MemoryMax=8G` and `MemoryHigh=6G` to cap total memory (server + all builders). Also changed `KillMode=process` → `KillMode=control-group` so stopping the service kills orphaned builders too.
2. **nq-tournament.service + timer** — masked (symlinked to /dev/null). Was crash-looping every 10 seconds due to `User=rob` directive in a user-level systemd unit (status=216/GROUP). 101K restarts eating CPU for nothing.

### Root Cause
The kanban runner spawns Claude Code / Codex as child processes. Each can load massive context + model weights. Two concurrent builders ate 24 GB with no memory cap. systemd's default is unlimited memory for user services.

### Prevention
- `MemoryMax=8G` hard cap on claw-kanban ensures systemd kills the builder (not the machine) if it exceeds 8 GB total
- `MemoryHigh=6G` triggers kernel memory pressure reclaim above 6 GB
- `KillMode=control-group` ensures no orphaned builder processes survive a service restart

## Blofin Data Architecture Investigation (12:25-12:50 PM MST)

### Problem
Rob said he had "all kinds of good data 1 min historical" working a month ago, but current implementation only stored close price. He told me to stop making stupid assumptions and investigate what data is ACTUALLY available before writing more code.

### What I Found

**WebSocket `tickers` channel (live data):**
- 13 fields per tick:
  - Price: `last`, `askPrice`, `bidPrice`
  - Size: `lastSize`, `askSize`, `bidSize`
  - 24h OHLC: `open24h`, `high24h`, `low24h`
  - 24h Volume: `vol24h` (contracts), `volCurrency24h` (base currency)
  - Timestamp: `ts`
  - Symbol: `instId`

**WebSocket `trades` channel (individual trades):**
- 6 fields: `tradeId`, `price`, `size`, `side` (buy/sell), `ts`, `instId`
- Rate: 100-1000 trades/sec during volatile periods
- **TOO HEAVY** — would generate 5-10M rows/hour

**REST `/market/candles` API (historical OHLCV):**
- 9 fields per 1-min candle:
  - `[timestamp, open, high, low, close, vol_contracts, vol_base, vol_quote, confirmed]`
- Rate limits: 20 req/2sec per endpoint
- Can fetch up to 100 candles per request
- Can paginate back 12+ months

### What I Was Doing Wrong
1. Only storing `close` price from candles (ignoring OHLCV)
2. Only storing `last` price from tickers (ignoring spread, volume, OHLC 24h)
3. Creating new Parquet schemas without checking what the old working version captured
4. Starting multiple backfill scripts without throttling properly

### The Right Architecture (documented in `brain/DATA_SOURCES_REAL.md`)

**Live data (WebSocket tickers):**
- Store all 13 fields in `data/tickers/YYYY-MM-DD.parquet`
- Used for: real-time signal detection, paper trading execution, dashboard live view
- Retention: 90 days rolling (auto-cleanup old dates)

**Historical OHLCV (REST candles):**
- Store 9 fields in `data/candles_1m/SYMBOL/YYYY-MM.parquet`
- Used for: strategy backtesting, model training, performance analysis
- Retention: 12 months (archive older data)
- Backfill: one-time historical load, then incremental daily updates

**Trades (NOT storing):**
- Too high volume (millions/hour during spikes)
- Aggregated data (tickers, candles) sufficient for our strategies

### What Got Fixed
1. Created `utils/parquet_ticker_writer.py` — stores all 13 ticker fields
2. Updated `ingestor_parquet.py` — calls `write_ticker()` with full data
3. Service restarted and verified capturing rich data ✅
4. Stopped old tick-only ingestor
5. Created proper backfill plan for historical candles (365 days)

### CRITICAL LESSON: INVESTIGATE BEFORE CODING
**Never assume what data is available.** Always:
1. Check API docs or make a test request first
2. Look at actual WebSocket messages (not just the schema)
3. Find the old working version and see what IT captured
4. THEN design the schema and writer

I wasted 30 minutes writing tick/candle/OHLCV writers without checking what data Blofin actually sends. Rob was right to stop me.
