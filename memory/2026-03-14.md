## Auto-Wake Cron Learning (Critical Operational Knowledge)

**Mistake:** Created Moonshot dashboard iteration cron with `sessionTarget: "main"` - tried to wake THIS session every 15 min for 3 hours.

**Why it failed:** Main session was continuously busy with church SMS investigation, so wake messages never came through. Cron showed `nextRunAtMs` but never executed.

**Correct approach:**
- `sessionTarget: "main"` → only works when main session is IDLE
- `sessionTarget: "isolated"` → spawns separate sessions, always works, runs independently

**Fixed cron:**
- Job ID: 114c7423-a7ba-4065-ac51-f6ef7a1061f2
- Uses isolated sessions
- Checks dashboard + Agent Team progress every 15 min
- Reports via delivery mode if needed

**Rule for future:** Use isolated sessions for ANY background/scheduled work that needs to run regardless of main session state.

## Church SMS Root Cause - REFINED (10:10-10:19 AM)

**Initial theory was WRONG:** Textbelt webhook infrastructure failure.

**ACTUAL ROOT CAUSE (Rob caught this):** The 18 manual SMS sends on March 13 from omen-claw did NOT include `replyWebhookUrl` parameter.

**Key insight:** Rob said "we sent those last 18 SMS manually from omen-claw!! none of them were from the cron!"

**How Textbelt webhooks work:**
- When sending via Textbelt API, you MUST include: `"replyWebhookUrl": "https://church-volunteer-coordinator.vercel.app/api/sms/incoming"`
- This tells Textbelt: "when recipient replies, POST it to this URL"
- If parameter is missing → Textbelt delivers the SMS but has nowhere to send replies → replies lost forever

**What happened on March 13:**
- 18 recruitment texts sent manually (probably via raw curl without webhook URL)
- 11 people replied → Textbelt had no webhook URL → replies disappeared
- Early morning replies (before 5 AM) worked because those were from OLDER texts that HAD webhook URL

**Solution created:**
- Proper send script: `/church-volunteer-coordinator/send_recruitment.sh`
- Always includes webhook URL in payload
- Usage: `./send_recruitment.sh "PHONE" "MESSAGE"`

**Signature verification issue (still pending):**
- Vercel endpoint requires `x-textbelt-signature` and `x-textbelt-timestamp` headers in production
- Unknown if Textbelt actually sends these headers
- Added detailed logging to endpoint to diagnose
- Added `DISABLE_SIGNATURE_CHECK` env var bypass option
- Test SMS sent at 10:18 AM with proper webhook URL included - waiting for reply to confirm

**Critical lesson:**
- When debugging, always verify BOTH directions of a flow (we initially only looked at Vercel endpoint, not at what was sent to Textbelt)
- Manual sends are dangerous - create scripts with proper defaults
- The person closest to the problem (Rob) often has the critical context
