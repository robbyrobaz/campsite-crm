## Token Usage OAuth Refresh — ACTUAL ROOT CAUSE & PERMANENT FIX (Mar 25 13:22)

**What I got wrong:**
- Assumed manual OAuth refresh via curl was the right approach
- Hit rate limits testing it aggressively (6-7 attempts in 10 min)
- Blamed the rate limit for blocking the fix
- Didn't realize Claude CLI has its own token refresh that works better

**What actually happened:**
- OAuth token in `~/.claude/.credentials.json` expired Mar 24 at 1:53 PM (normal expiry)
- Gateway uses DIFFERENT token from `auth-profiles.json` (why conversations still worked)
- Usage API needs the OAuth token from credentials.json with `user:profile` scope
- Running `claude --version` does NOT trigger token refresh
- Running actual inference (`echo "test" | claude`) DOES trigger refresh
- Claude CLI updates credentials.json automatically when it refreshes

**The REAL fix (deployed Mar 25 13:22):**
```python
def _refresh_oauth_token_via_cli(creds_path):
    """Refresh by triggering Claude CLI to do it."""
    old_mtime = os.path.getmtime(creds_path)
    subprocess.run(["bash", "-c", "echo 'ping' | timeout 10 claude 2>/dev/null"], ...)
    new_mtime = os.path.getmtime(creds_path)
    return new_mtime > old_mtime  # True if file was updated
```

**Key insight:** Don't reinvent OAuth flows. Use the CLI's built-in refresh by making a minimal API call.

**File:** `/home/rob/.openclaw/workspace/brain/scripts/token_usage_aggregator.py`

**Verified working:** Fresh data at 13:22 (5h: 9%, 7d: 75%, 7d Sonnet: 64%)

**Lesson:** When a tool already has working auth, trigger it instead of reimplementing it.
