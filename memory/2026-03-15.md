## OOM Crash & Hard Reboot (~8:29 AM MST)

### What Happened
1. 08:29:33 — OOM kill: python3 (pid 3374158) from claw-kanban.service — 13.3 GB RSS
2. 08:33:30 — OOM kill: python3 (pid 3373705) from claw-kanban.service — 10.3 GB RSS
3. 24 GB combined from kanban builder processes on 32 GB machine → cascading OOM
4. systemd-journald and systemd-resolved crashed → unresponsive → hard reboot at ~08:35
5. Also: nq-tournament.service crash-looping (101,176 restarts!) with "Failed to determine supplementary groups" — `User=rob/Group=rob` in a user-level unit (not allowed)

### Fixes Applied
1. **claw-kanban.service** — added `MemoryMax=8G` and `MemoryHigh=6G` to cap total memory (server + all builders). Also changed `KillMode=process` → `KillMode=control-group` so stopping the service kills orphaned builders too.
2. **nq-tournament.service + timer** — masked (symlinked to /dev/null). Was crash-looping every 10 seconds due to `User=rob` directive in a user-level systemd unit (status=216/GROUP). 101K restarts eating CPU for nothing.

### Root Cause
The kanban runner spawns Claude Code / Codex as child processes. Each can load massive context + model weights. Two concurrent builders ate 24 GB with no memory cap. systemd's default is unlimited memory for user services.

### Prevention
- `MemoryMax=8G` hard cap on claw-kanban ensures systemd kills the builder (not the machine) if it exceeds 8 GB total
- `MemoryHigh=6G` triggers kernel memory pressure reclaim above 6 GB
- `KillMode=control-group` ensures no orphaned builder processes survive a service restart

## Blofin Stack Parquet Migration (9:32-9:58 AM MST, 26 minutes)

### Problem
- 24 GB SQLite DB on HDD
- Ingestor + paper engine + backtester fighting for DB lock
- "DB locked" warnings weekly
- Websocket reconnects every ~2 min (Blofin keepalive timeout)

### Solution: DuckDB + Parquet
**Old:** WebSocket → SQLite (24 GB, HDD)  
**New:** WebSocket → Parquet (12x compression, NVMe) → DuckDB reads

### Implementation
1. **Phase 1 (9:32):** New `ingestor_parquet.py` started (Parquet writer, 10K batch, 10s flush)
2. **Phase 2 (9:39):** Created `utils/duckdb_tick_adapter.py` (SQLite-compatible Parquet query wrapper)
3. **Phase 3 (9:39):** Updated `paper_engine.py` to read ticks from Parquet via DuckDB
4. **Phase 4 (9:56):** Stopped old SQLite ingestor
5. **Phase 5:** 24h verification (cron fires 2026-03-16 09:58)

### Results (First 30 min)
- **Ticks ingested:** 1.63M (~880 ticks/sec sustained)
- **Storage:** 18 MB Parquet vs ~5-10 GB SQLite (12x compression)
- **Paper engine:** Zero errors, reading from Parquet
- **Old ingestor:** STOPPED
- **Old DB:** Kept on HDD for `strategy_registry` metadata (read-only)

### Architecture
```
NVMe SSD:
  data/ticks/*.parquet          ← Rolling 90-day window (~30GB projected)
  data/paper_trading.db         ← Live paper trades (<500MB)

HDD (/mnt/data):
  blofin_monitor.db             ← OLD 24GB DB (strategy_registry, read-only)
  backtest_results.db           ← Strategy backtest results (~2GB/year)
```

### Files
- `utils/parquet_tick_writer.py` — batched Parquet append
- `ingestor_parquet.py` — WebSocket → Parquet
- `utils/duckdb_tick_adapter.py` — SQLite-compatible DuckDB wrapper
- `brain/DUCKDB_ARCHITECTURE.md` — full design doc

### CRITICAL LESSON: COPY-VERIFY-DELETE
**NEVER move data across filesystems.** Lost 107 GB (3 weeks of tick data) on Mar 12 by using `mv` instead of `cp`. Always:
1. COPY to destination
2. VERIFY copy is complete and correct
3. DELETE source only after verification
