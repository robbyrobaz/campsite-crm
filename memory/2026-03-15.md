## OOM Crash & Hard Reboot (~8:29 AM MST)

### What Happened
1. 08:29:33 — OOM kill: python3 (pid 3374158) from claw-kanban.service — 13.3 GB RSS
2. 08:33:30 — OOM kill: python3 (pid 3373705) from claw-kanban.service — 10.3 GB RSS
3. 24 GB combined from kanban builder processes on 32 GB machine → cascading OOM
4. systemd-journald and systemd-resolved crashed → unresponsive → hard reboot at ~08:35
5. Also: nq-tournament.service crash-looping (101,176 restarts!) with "Failed to determine supplementary groups" — `User=rob/Group=rob` in a user-level unit (not allowed)

### Fixes Applied
1. **claw-kanban.service** — added `MemoryMax=8G` and `MemoryHigh=6G` to cap total memory (server + all builders). Also changed `KillMode=process` → `KillMode=control-group` so stopping the service kills orphaned builders too.
2. **nq-tournament.service + timer** — masked (symlinked to /dev/null). Was crash-looping every 10 seconds due to `User=rob` directive in a user-level systemd unit (status=216/GROUP). 101K restarts eating CPU for nothing.

### Root Cause
The kanban runner spawns Claude Code / Codex as child processes. Each can load massive context + model weights. Two concurrent builders ate 24 GB with no memory cap. systemd's default is unlimited memory for user services.

### Prevention
- `MemoryMax=8G` hard cap on claw-kanban ensures systemd kills the builder (not the machine) if it exceeds 8 GB total
- `MemoryHigh=6G` triggers kernel memory pressure reclaim above 6 GB
- `KillMode=control-group` ensures no orphaned builder processes survive a service restart
